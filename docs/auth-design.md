# Authentication System Design

This document describes the implemented authentication architecture for BitMerchant.

## Goals

- Passkey-first authentication for owners and staff.
- Organization-style access control for multi-restaurant support.
- Role-based permissions for owner and kitchen workflows.
- Preserve anonymous customer ordering/session behavior.

## Core Concepts

### User

`internal/domain/user.go`

- Represents an authenticated actor.
- Stores WebAuthn credentials (`[]webauthn.Credential`).
- Implements `webauthn.User` interface required by `go-webauthn/webauthn`.

### Membership

`internal/domain/membership.go`

- Links a `User` to a `Restaurant` with a role.
- Supported roles:
  - `owner`
  - `kitchen_staff`
  - `customer`

### Invitation

`internal/domain/invitation.go`

- Owner can issue invitation tokens for staff onboarding.
- Invitations are single-use and time-bound.

### Session

`internal/domain/session.go`

- Server-side session object keyed by `bitmerchant_session` cookie.
- Can be anonymous (`UserID == nil`) or authenticated.
- Carries active `RestaurantID` context used by protected routes.

## Persistence Interfaces

`internal/domain/repositories.go` now includes:

- `UserRepository`
- `MembershipRepository`
- `InvitationRepository`
- `SessionRepository`

In-memory implementations are provided under:
`internal/infrastructure/repositories/memory/`.

## WebAuthn Integration

`internal/infrastructure/auth/webauthn.go`

- Wraps `go-webauthn/webauthn`.
- Configured via RP ID / origin from `BASE_URL`.
- Supports:
  - `BeginRegistration` / `FinishRegistration`
  - `BeginPasskeyLogin` / `FinishPasskeyLogin`
- Stores short-lived ceremony state by session ID.

## HTTP Layer

### Auth Handler

`internal/interfaces/http/auth.go`

Endpoints:

- `GET /auth/signup`
- `GET /auth/login`
- `GET /auth/invite/:token`
- `POST /auth/register/begin`
- `POST /auth/register/finish`
- `POST /auth/login/begin`
- `POST /auth/login/finish`
- `POST /auth/logout`
- `GET /auth/select-restaurant`
- `POST /auth/select-restaurant`
- `POST /dashboard/invite`

Behavior:

- Signup creates user + restaurant + owner membership.
- Invite flow creates user + membership for invited restaurant/role.
- Login uses discoverable passkey assertions.
- Login behavior is context-aware:
  - if user has one membership, restaurant context is set automatically
  - if user has multiple memberships, user is redirected to `/auth/select-restaurant`
- Session ID is rotated on successful login/registration to reduce fixation risk.
- Invitation updates are now enforced (fail closed when token-consumption persistence fails).
- Restaurant selection updates the server-side session `RestaurantID`, which is then enforced by `RequireRole(...)`.

### Middleware

`internal/interfaces/http/middleware/`

- `SessionMiddlewareWithReposAndOptions(...)`:
  - Creates/reads cookie session.
  - Loads authenticated user/session into context when available.
  - Supports secure-cookie and TTL options.
- `RequireAuth()`:
  - Redirects unauthenticated requests to `/auth/login`.
- `RequireRole(...)`:
  - Enforces role membership for active restaurant context.

`SessionMiddleware()` and `SessionMiddlewareWithRepos(...)` are kept for backward compatibility in tests and legacy setup code.

## Template + Browser Integration

New templates:

- `internal/interfaces/templates/auth_signup.templ`
- `internal/interfaces/templates/auth_login.templ`
- `internal/interfaces/templates/auth_invite.templ`
- `internal/interfaces/templates/auth_select_restaurant.templ`

Generated files:

- `*_templ.go` variants generated by `templ generate`.

Client script:

- `static/js/passkey.js`

Responsibilities:

- Fetch begin options.
- Convert base64url <-> ArrayBuffer for browser APIs.
- Execute `navigator.credentials.create/get`.
- Submit finish payload and redirect based on response.

## Route Protection

`cmd/server/main.go` now wires protected groups:

- `/dashboard/*` -> owner only
- `/admin/*` -> owner only
- `/kitchen/*` -> owner or kitchen staff
- `/auth/select-restaurant` -> authenticated users only

Handlers for admin/dashboard/kitchen now read restaurant context from authenticated session first.
If restaurant context is missing on protected operations, handlers now fail instead of using an implicit tenant fallback.

## Persistence Notes

When `DATABASE_URL` is configured, both auth and core business repositories are backed by PostgreSQL:

- auth: users, memberships, invitations, sessions
- core: restaurants, menu categories, menu items, orders, order items, payments

Goose migrations are applied at startup before repository usage.

## Hardening Notes

- Existing anonymous customer flow remains intact.
- WebAuthn ceremony state now has stricter lifecycle checks:
  - typed ceremony matching (registration vs login)
  - explicit missing-key checks
  - expiry checks (library expiry + 5 minute max ceremony age)
- Auth handler responses are sanitized for client-facing errors while logging structured details server-side.
- Logout explicitly expires the session cookie in the browser.

## Security Defaults

- Session cookie: `HttpOnly`, `SameSite=Strict`, 24h expiry.
- Secure cookie: enabled automatically when `BASE_URL` is HTTPS, or forced with `COOKIE_SECURE=true`.
- Invitation tokens: random 32-byte URL-safe values.
- Invitation expiry: 7 days.
- CSRF middleware remains active for state-changing requests.
